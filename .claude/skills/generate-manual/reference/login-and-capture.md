# 로그인 · 화면 캡처 상세 레퍼런스

`generate-manual` 스킬이 브라우저를 구동할 때 참고하는 셀렉터·상호작용 식별 규칙. SKILL.md 절차의 부록이다. 캡처는 모두 `chrome-devtools-mcp` 도구로 수행한다(별도 자동화 라이브러리 도입 금지).

---

## 1. 로그인 흐름

근거: `apps/host/src/app/pages/Login.tsx`.

로그인 페이지(`{baseUrl}/login`)는 Ant Design Form이며 입력 필드는 **placeholder**로 식별한다.

| 항목 | 식별값 | 비고 |
| --- | --- | --- |
| 아이디 | placeholder `아이디` (`name="userAccount"`) | 필수 |
| 비밀번호 | placeholder `비밀번호` (`name="password"`, Input.Password) | 필수 |
| 테넌트명 | placeholder `테넌트명 (멀티테넌트 사용자만 입력)` (`name="tenant"`) | 멀티테넌트 계정만 |
| 로그인 버튼 | 버튼 텍스트 `로그인` (`htmlType="submit"`) | 클릭으로 제출 |

### 절차
1. `navigate_page` → `{baseUrl}/login`. `take_snapshot`으로 폼 노출 확인.
2. `fill` 또는 `fill_form`으로 아이디/비밀번호(+테넌트가 설정에 있으면 테넌트) 입력.
3. `로그인` 버튼 `click`.
4. 결과 분기 처리:
   - **성공**: URL이 `/`(또는 메인)로 이동. `wait_for`로 레이아웃(사이드바/헤더) 노출 확인 후 진행.
   - **테넌트 선택 다이얼로그**(`tenant_required`): 멀티테넌트 계정이면 `TenantSelectionDialog`가 뜬다. 설정의 `login.tenant` 값과 일치하는 항목을 클릭. 설정에 테넌트가 없으면 **사용자에게 보고 후 중단**.
   - **비밀번호 변경 다이얼로그**(`password_change_required` / 만료): `ChangePasswordDialog`가 강제로 뜬다. 자동 처리하지 말고 **사용자에게 보고 후 중단**(비밀번호 변경은 부수효과가 크므로).
   - **로그인 실패 토스트**(아이디/비밀번호 오류, 계정 잠금 등): 메시지를 그대로 사용자에게 전달하고 중단.

### 세션 유지
로그인 성공 후 세션은 브라우저 컨텍스트(쿠키/스토리지)에 유지된다. 같은 페이지 세션에서 각 화면 URL로 `navigate_page`만 하면 재로그인이 필요 없다. 캡처 도중 `/login`으로 튕기면 세션이 만료된 것이므로 재로그인 후 해당 라우트부터 재개한다.

## 2. 화면 URL 구성

- 전체 URL = `{baseUrl}/{appId}{routes.tsx leaf path}` (host가 `/manager`, `/fca` … 로 각 remote를 마운트. 근거: `apps/host/src/app/app.tsx`).
- 정확한 목록은 `extract-routes.mjs` 산출 매니페스트의 `fullUrl`을 사용한다(직접 조합하지 말 것).

## 3. 화면별 상호작용 식별 (코드 정독 기반)

각 라우트의 `componentFile`(매니페스트 제공)과 그 페이지가 import하는 `features/<feature>/...`를 읽어 아래 요소를 식별하고, 화면에서 실제로 조작하며 캡처한다. 요소가 없거나 데이터가 없으면 graceful skip 후 매뉴얼에 사유를 1줄 표기한다.

### 3-1. 탭 (Ant Design Tabs / shadcn Tabs)
- 코드 신호: `<Tabs items={...} />`, `items: [{ key, label, children }]`, 또는 `*DetailLayout` 컴포넌트(예: fca `ModelDetailLayout`).
- 캡처: 각 탭 라벨을 순서대로 `click` → `wait_for` → 스크린샷. 파일명 상태 접미사 `__tab-<label>`.
- **모든 탭을 빠짐없이 캡처한다 — "대표 탭만" 캡처 금지**: 탭이 N개면 스크린샷도 N장이어야 한다(기본 활성 탭은 `__base`/`__detail`로, 나머지는 각각 `__tab-<label>`/`__detail-tab-<label>`로). 시간을 아끼려고 일부 탭을 건너뛰지 말 것. 빠른 자가검증: 캡처 종료 후 그 상세의 탭 개수와 해당 상세의 이미지 장수(기본 1 + 탭 전환분)가 일치하는지 확인한다.
- **같은 컴포넌트를 쓰더라도 경로(`fullUrl`)가 다르면 별개 화면으로 캡처한다 — "다른 화면과 동일함" 참조로 대체 금지**: 매니페스트의 fullUrl이 다르면(예: `bot-config/model/...` vs `global/model/...`) 운영자가 보는 breadcrumb·메뉴 맥락·헤더가 달라 **사실상 다른 화면**이다. 컴포넌트가 동일하다는 이유로 한쪽을 "X와 같습니다"로 퉁치지 말고, 각 경로마다 자체 스크린샷(기본 + 모든 탭 + 진입 가능한 하위 상세)을 찍는다. 설명 문장은 유사 화면을 가리켜 간결히 써도 되지만, **이미지는 각 경로별로 따로** 둔다.
- **⚠️ 탭 전환은 반드시 chrome-devtools-mcp의 `click` 도구(uid 기반 실제 클릭)를 쓴다.** `evaluate_script`로 `element.click()`(합성 클릭)을 호출하면 **shadcn/Radix Tabs는 전환되지 않는다**(Radix 트리거는 click이 아니라 pointerdown으로 활성화됨). 합성 클릭을 쓰면 매 탭마다 첫 탭 화면만 반복 캡처되어 **파일명만 다르고 그림이 전부 같은** 결과가 나온다(Ant Design Tabs는 합성 click에도 전환되지만, 구분 없이 `click` 도구로 통일할 것).
- **탭별 캡처 직후 검증**: 같은 상세의 탭 이미지들이 서로 다른지 확인한다(예: 캡처 종료 후 `md5sum {appId}/images/<slug>__tab-*.png` 비교 → 고유 개수 == 탭 개수). 동일 해시가 나오면 전환 실패이므로 `click` 도구로 재캡처한다.

### 3-2. 목록 → 상세 진입 (AG-Grid 등)
- 동적 상세(`dynamic: true`, `enterFromUrl` 보유)는 직접 `:id` URL로 못 가므로, `enterFromUrl`(목록)로 이동 → **첫 번째 행 클릭** → 상세 진입.
- AG-Grid 행: snapshot에서 첫 데이터 행(`.ag-row[row-index="0"]` 영역의 셀) 클릭. 행 클릭으로 상세 이동이 안 되면 행 액션 버튼(상세/편집 아이콘)을 찾아 클릭.
- 진입 후 상세 화면 + (있으면) 상세 탭까지 캡처.
- **데이터가 0건**이면 상세 캡처를 스킵하고 매뉴얼에 "예시 데이터가 없어 상세 화면은 생략" 표기.

### 3-3. 생성/편집 드로어·모달
- 코드 신호: `forwardRef` + `useImperativeHandle`로 노출된 Drawer/Modal ref(프로젝트 add-drawer 패턴), 부모에서 `ref.current?.open(...)` 호출.
- 트리거: 화면 상단 우측의 주요 버튼(텍스트 휴리스틱 `추가`, `등록`, `생성`, `편집`, `신규`). `click` → `wait_for`로 드로어/모달 노출 확인 → 스크린샷(`__create` / `__edit`) → 닫기(`취소`/`닫기`/X 버튼 또는 ESC).
- 편집 드로어는 목록 행의 편집 액션 또는 상세 화면의 `편집` 버튼에서 연다.

### 3-4. 검색·필터
- 코드 신호: 페이지 상단 `header` 내 `Select`/`Input`/`Collapsible` 필터.
- 기본 상태 스크린샷에 이미 포함되므로 별도 조작은 생략하되, 필터 항목의 **의미**는 설명 텍스트에 반영한다.

### 3-5. 현재 화면명 확인
- breadcrumb은 host SubHeader가 그린다(페이지 본문엔 없음). snapshot의 상단 breadcrumb 텍스트로 화면명을 확인해 매뉴얼 제목·설명에 활용한다.

### 3-6. 개발 서버 오류 오버레이(`removeChild`) 처리 — 다시 캡처

캡처 도중(특히 라우트 이동·탭 전환·상세 진입 직후) 화면 전체가 검게 덮이며 다음과 같은 **개발 서버 전용 오류 오버레이**가 뜰 때가 있다.

```
Uncaught runtime errors:
ERROR
Cannot read properties of null (reading 'removeChild')
TypeError: Cannot read properties of null (reading 'removeChild')
    at HTMLLinkElement.<anonymous> (http://localhost:4200/styles.js:...)
```

- **성격**: dev 서버의 스타일 핫리로드(HMR)가 `<link>` 정리 과정에서 내는 **개발 모드 한정 일시 오류**다(webpack 시절 관측 — rsbuild 전환 후에도 유사 오버레이가 뜨면 동일하게 처리). 실제 화면 동작·라우팅과는 무관하며(보통 그 아래에서 목표 화면으로의 이동은 이미 성공해 있다), 운영 빌드에는 나타나지 않는다. **매뉴얼에 이 오류를 절대 노출하지 말 것**(스크린샷에 검은 오버레이가 찍히면 그 컷은 폐기).
- **처리**: 오버레이가 화면을 가린 채 스크린샷이 찍혔으면 `navigate_page` `type: "reload"`로 **새로고침한 뒤 다시 캡처**한다. 새로고침하면 오버레이가 사라지고 깨끗한 화면이 나온다. (오버레이의 `Dismiss` 버튼을 눌러도 되지만, iframe이 여러 개 겹쳐 뜨는 경우가 있어 reload가 더 확실하다.)
- **검증**: 동적 상세(`:id`)에서 발생하면 reload 후 URL이 의도한 상세 경로(예: `.../entity/<id>`)인지 확인한다. 가끔 오류로 진입 자체가 막혀 목록/부모로 돌아가 있을 수 있는데, 그때는 해당 진입 동작(행 더블클릭 등)을 한 번 더 수행한 뒤 다시 캡처한다.
- **사후 점검**: 앱 캡처를 마치면 산출 이미지를 한 번 훑어 **검은 오류 화면이 섞이지 않았는지** 확인하고, 있으면 그 화면만 reload→재캡처한다.

## 4. 스크린샷 저장 규칙

- 저장 경로: `{outputDir}/{appId}/images/`.
- 파일명: `<route-slug>__<state>.png`.
  - `route-slug`: `fullUrl`에서 `/{appId}/` 이후를 `-`로 치환(예: `bot-config/bot/list` → `bot-config-bot-list`). 동적 세그먼트 `:x`는 `_x`로.
  - `state`: `base`(기본), `tab-<label>`, `detail`, `detail-tab-<label>`, `create`, `edit`.
- 뷰포트는 설정 `viewport`(기본 1920×1080)로 `resize_page` 후 캡처해 해상도를 통일한다.
- 전체 페이지가 길면 `take_screenshot`의 fullPage 옵션을 사용한다.

## 5. 설명 작성 원칙 (제품 설명서 톤)

대상은 **운영자/클라이언트**다. 실제 제품 사용 설명서를 쓴다는 마음으로, 군더더기 없는 자연스러운 존댓말 문장으로 작성한다.

### 5-0. 톤과 구조

- **정형화된 질문형 소제목 금지**: `이 화면은 무엇인가`, `무엇을 할 수 있는가`, `단계별 사용법`, `화면 항목 설명` 같은 템플릿 라벨을 그대로 박지 않는다. 제품 설명서에는 이런 메타 문구가 어울리지 않는다.
- **권장 섹션 구조** (화면마다 필요한 것만):
  1. **제목** — breadcrumb의 화면명.
  2. **스크린샷** — 제목 바로 아래(상태가 여러 개면 해당 설명 가까이 추가 배치).
  3. **설명 문단** — 이 화면의 역할과 핵심 동작을 한두 문단으로 **녹여서** 쓴다. "무엇을 보는 화면이고, 여기서 무엇을 할 수 있는지"를 끊어진 항목이 아니라 흐르는 문장으로. 예: "등록된 봇을 카드 형태로 모아 보여주는 화면입니다. 봇마다 버전·연결된 모델·대화 수를 한눈에 확인하고, 새 봇을 추가하거나 기존 봇을 열어 설정을 바꿀 수 있습니다."
  4. **`사용 방법`** (선택) — 대표 작업의 절차가 비자명할 때만 번호 목록으로. 누르면 바로 아는 동작까지 일일이 단계로 늘어놓지 않는다.
  5. **`화면 항목`** (선택) — 검색 필터·표 항목·입력 항목이 많아 표가 도움이 될 때만 표로 정리. 적으면 설명 문단에 자연히 녹인다.
- **고지·면책·제작 메모 금지**: "권한·환경에 따라 다를 수 있음", "예시 데이터가 없어 생략", "알려진 한계", "최고 권한 계정 권장" 등 **제작 과정/제약을 알리는 문구를 본문(서두 포함)에 넣지 않는다**. 데이터가 비어 있어도 그 화면의 정상 용도·기능을 담담히 기술한다.
- 개발 용어(`props`·`컴포넌트`·`훅`·`API`·remote·Variant·드로어·그리드/컬럼 등)는 5-1 매핑대로 순화한다.
- 코드에서 읽은 라벨·옵션·상태값(상수 객체의 `_LABELS` 매핑 등)을 근거로 정확히 기술하되, 내부 enum/키 값이 아니라 사용자에게 보이는 한글 라벨을 쓴다.

### 5-1. 금지어 → 권장 표현 매핑 (개발 용어 순화)

운영자/클라이언트는 프론트엔드·인프라 용어를 모른다. 아래 단어들은 **본문(제목·설명·이미지 alt 포함)에서 쓰지 말고** 권장 표현으로 바꾼다. 목록은 예시이며, 같은 부류의 개발 용어가 나오면 동일 원칙으로 순화한다.

| 금지(개발 용어) | 권장 표현 |
| --- | --- |
| remote / 마이크로 프론트엔드 | 화면 모음 (또는 그냥 생략) |
| Variant / 페이지 Variant / 변형 | 화면 지정 / 맞춤 화면 |
| BFF Flow | API 경로 |
| OAuth (클라이언트) | 인증용 클라이언트 |
| 그리드 / 컬럼 | 표 / 항목 (표 항목) |
| 드로어(Drawer) | 패널 / 입력창 / 슬라이드 패널 |
| 모달 | 팝업 창 |
| 마법사(Wizard) | 단계별 입력 (예: "두 단계에 걸쳐 입력") |
| 라우트 / 경로(route) | 화면 |
| 컴포넌트 / props / 훅 / 렌더 | (쓰지 않음 — 동작·항목으로 설명) |

**예외 — 화면에 실제로 보이는 라벨은 유지한다.** 화면 버튼·헤더·필드에 그대로 노출되는 약어(NLU·STT·TTS·AOE·UCID·DNIS·RTP·Export·Grant Types 등)는 운영자가 화면에서 보는 단어이므로 **그대로 쓰되, 한 문서에서 처음 나올 때 한 번 풀어서 설명**한다(예: `언어 이해(NLU) 모델`, `음성을 텍스트로 변환(STT)`, `통화식별자(UCID)`, `Grant Types(허용 방식)`). 노드 설정의 NAT·DR·IC 등 담당자 전용 설정 항목명도 화면 라벨이면 유지한다.

> 작성 후 `remote|Variant|BFF|OAuth|컬럼|그리드|마법사|드로어` 등으로 본문을 한 번 grep해 잔존 개발 용어가 없는지 점검한다.
