# commit

이 저장소의 커밋 메시지 작성 절차. 사용자가 "커밋 메시지", "커밋해줘", "commit" 등을 요청하면 이 절차를 따른다.

## 출력 원칙

- **커밋 메시지를 먼저 제시하고 사용자 확인을 기다린다.** 사용자가 "커밋해", "실행해" 등 명시적으로 요청하기 전까지 `git commit`을 실행하지 않는다.
- 메시지는 코드 블록으로 출력한다.
- 제목만으로 끝내지 말고 반드시 본문(body)을 포함한다.

## 절차

### 1. 변경 사항 확인 (추측 금지)

아래를 실행해 실제 변경 내용을 직접 확인한다:

```bash
git status
git diff --staged   # 스테이징된 변경
git diff            # 스테이징 안 된 변경
git log --oneline -10   # 최근 커밋 스타일 참고
```

변경된 파일 목록, 실제 diff, 최근 커밋의 톤을 눈으로 확인한 뒤 아래 단계로 진행. 추측으로 메시지를 쓰지 않는다.

### 2. 카테고리 판정

**변경 내용의 실질적 영향**을 기준으로 선택한다. 단순히 파일 위치나 키워드로 판단하지 않는다.

| 카테고리 | 용도 |
| --- | --- |
| 🎉 init | 프로젝트/모듈 초기화 |
| ✨ feat | 새로운 기능 추가 |
| 🐛 fix | 버그 수정 (동작 오류 해결) |
| 💄 design | 스타일/레이아웃/정렬 등 시각적 변경 |
| 🔨 refactor | 동작 변화 없는 구조 개선 |
| 📦️ chore | 빌드/설정/도구 변경 |
| ✅ test | 테스트 추가/수정 |
| 🚀 deploy | 배포 관련 |
| 🚚 rename | 파일/폴더명 변경 |
| 📚 docs | 문서 변경 |
| 🔥 remove | 코드/파일 제거 |

판단이 애매하면 diff를 다시 보고 "이 변경이 실제로 무엇을 해결/달성하는가"로 되짚어본다.

### 3. Scope 판정

**scope는 반드시 remote 앱 명칭**(예: `apps/fca` → `fca`). feature 폴더명(`iam`, `bot-config` 등)을 scope로 쓰지 않는다.

판정 규칙:

1. 변경된 파일 경로를 수집한다.
2. 모든 변경이 **단일 `apps/<remote>/` 하위**에 있다 → scope는 그 `<remote>`.
3. 변경이 **여러 remote에 걸쳐 있다** → scope 생략.
4. 변경이 **`libs/` 하위만 있다** → scope 생략.
5. 변경이 **`apps/`와 `libs/`에 걸쳐 있다** → scope 생략.
6. 스크립트/설정 등 remote에 속하지 않는 변경 → scope 생략.

유효한 remote 목록은 **하드코딩하지 말고** `apps/` 디렉토리를 직접 확인한다(`ls apps/`). remote는 계속 추가/변경될 수 있다.

### 4. 메시지 작성

**포맷:**

```
<emoji><type>(<scope>): <제목>

<본문>
```

- scope가 없을 때: `<emoji><type>: <제목>` (괄호 생략)
- 제목: 간결하게, 변경의 핵심만 (~50자 내외)
- 본문: 변경 사항의 상세 내용. 무엇을/왜 바꿨는지를 기술. 제목만으로 커밋하지 않는다.
- 이모지와 타입 사이에 공백 없음 (`✨feat`), 타입과 scope 사이 공백 없음 (`feat(fca)`).

### 5. 제시 및 대기

작성한 메시지를 코드 블록으로 출력하고 사용자 확인을 기다린다. 사용자가 수정을 요청하면 반영 후 다시 제시.

**사용자가 명시적으로 "커밋 실행" / "커밋해줘" 등을 요청한 경우에만** 다음 단계로 진행:

```bash
git commit -m "$(cat <<'EOF'
<emoji><type>(<scope>): <제목>

<본문>
EOF
)"
```

- 스테이징되지 않은 변경만 있다면 먼저 사용자에게 어떤 파일을 `git add`할지 확인한다.
- `git add -A`, `git add .` 사용 금지 (민감 파일 실수 방지). 파일명을 명시해 add.
- `--no-verify` 등 훅 우회 금지. 훅 실패 시 원인 수정 후 다시 시도.
- `--amend` 사용 금지 (사용자가 명시 요청한 경우 제외).

## 체크리스트

메시지 제시 전 점검:

- [ ] `git diff`로 실제 변경을 눈으로 확인했는가?
- [ ] 카테고리가 변경의 실질적 영향과 일치하는가?
- [ ] scope가 remote 명칭인가? (feature명 아님)
- [ ] 여러 remote 걸친 변경이면 scope를 뺐는가?
- [ ] 제목 + 본문을 모두 포함했는가?
