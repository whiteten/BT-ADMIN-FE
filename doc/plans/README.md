# doc/plans — RFC(변경 제안서)

**큰 작업이나 프로젝트에 영향이 큰 변경은, 착수 전에 이 폴더에 RFC(변경 제안서)를 먼저 작성하고 팀과 공유한 뒤 진행합니다.**

여기서 RFC는 "큰 변경을 착수 전에 문서로 정리해 팀에 공유하는 것"을 뜻합니다. 승인을 받아야 진행할 수 있는 게이트가 아니라, 방향과 근거를 미리 알리는 문서입니다.

혼자 판단해서 바로 코드부터 손대면, 다른 팀원의 로컬 환경·CI·다른 작업과 충돌하거나, 되돌리기 어려운 결정이 아무도 모르게 박혀버립니다. RFC를 먼저 세우면 (1) 다른 사람이 이어받거나 검토할 수 있고, (2) 결정 근거와 대안이 기록으로 남고, (3) 착수 사실과 방향을 팀이 미리 알 수 있습니다.

## 언제 RFC가 필요한가 (트리거)

아래 중 하나라도 해당하면 **작성 대상**입니다:

- **되돌리기 어려운 결정** — 아키텍처 변경, 데이터·API 계약 변경, 마이그레이션(포기 시 원복 비용이 큰 것)
- **모두의 환경에 영향** — 의존성 메이저 업그레이드, 빌드 도구·번들러·turborepo·Module Federation 구조 변경, `pnpm-lock.yaml`·tsconfig·공통 설정 변경 등 팀원 로컬·CI가 함께 바뀌어야 하는 것
- **넓은 변경 범위** — 여러 remote 앱을 동시에 건드리거나, `libs/shared-*` 공유 라이브러리의 계약을 바꾸는 것
- **긴 호흡** — 한 번에 못 끝내고 여러 세션·여러 명이 이어받아야 하는 것
- **불확실성이 큼** — 실현성 조사·성능 실측·대안 비교가 결론에 필요한 것

반대로 **불필요한 경우**: 단일 화면·기능 추가, 버그 수정, 소규모 리팩터, 한 앱 안에서 끝나고 되돌리기 쉬운 변경. 이런 건 그냥 진행하고 커밋 메시지로 충분합니다.

> 애매하면 기준은 하나: **"내가 착수하기 전에 다른 팀원이 알아야 하나?"** 예면 RFC를 쓰세요.

## 절차

1. 착수 **전에** 이 폴더에 RFC를 만든다(아래 폴더 규칙 참조).
2. 팀과 공유한다(피드백은 받으면 좋지만, 진행을 막는 승인 절차는 아니다).
3. 계획대로 진행한다. 진행 중 바뀐 결정·발견은 같은 문서에 갱신한다(RFC는 살아있는 문서).
4. 완료 후에도 문서는 남긴다 — 왜 그렇게 했는지의 근거로.

## 폴더 규칙

```
doc/plans/
├── README.md                 ← 이 파일
├── platform/                 ← 빌드·번들러·인프라·의존성·마이그레이션 등 플랫폼 차원
│   └── <이니셔티브>/          ← 관련 문서를 이니셔티브별 하위폴더로 묶음
│       ├── INDEX.md          ← 그 폴더의 문서 목록·한줄요약
│       └── <RFC>.md
└── feature/                  ← 기능 단위의 큰 작업(여러 화면·여러 앱에 걸치는 기능 등)
    └── <이니셔티브>/
```

- **`platform/` vs `feature/`**: 플랫폼·인프라 성격이면 `platform/`, 제품 기능 성격이면 `feature/`.
- **이니셔티브별 하위폴더**: 관련 문서(조사·계획·측정 등)를 한 폴더로 묶어 상대링크로 연결한다. 루트에 파일을 흩뿌리지 않는다.
- **`INDEX.md` 유지**: 각 이니셔티브 폴더에 `INDEX.md`를 두고 문서를 추가·수정할 때마다 한 줄(`- [제목](파일명) — 한줄 설명`)을 갱신한다.
- **파일명**: kebab-case, **날짜 프리픽스 없음**(생성·수정 이력은 git이 관리). 예: `migration-plan.md`.
- **RFC 안에는** 배경·목적, 확정 결정, 대안·근거, 단계 계획, 검증 방법, 롤백, 진행 로그를 담아 **다른 세션·다른 사람이 이 문서만으로 이어받을 수 있게** 쓴다. 레퍼런스: [platform/turborepo-rsbuild-migration/](platform/turborepo-rsbuild-migration/).
